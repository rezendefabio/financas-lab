## Tarefas

### Tarefa 1 — Criar a migration V1

**Caminho:** `src/main/resources/db/migration/V1__schema_inicial.sql`

**Conteúdo exato:**

```sql
-- V1__schema_inicial.sql
--
-- Migration inicial do schema. Cria a tabela placeholder __healthcheck que
-- existe apenas para validar que o pipeline Flyway -> Postgres -> Spring
-- funciona end-to-end. Pode ser referenciada pelo endpoint de healthcheck
-- da Etapa 2.3.
--
-- Convenção: tabelas com prefixo `__` (dois underscores) são tecnicas/
-- internas, nao representam dominio de negocio.

CREATE TABLE __healthcheck (
    id          BIGSERIAL PRIMARY KEY,
    verificado_em TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

-- Insere uma linha placeholder para que SELECT count(*) > 0 seja sempre
-- verdadeiro mesmo sem inserts da aplicacao.
INSERT INTO __healthcheck DEFAULT VALUES;
```

Razoes das escolhas (nao vao no commit, sao pra contexto):

- `BIGSERIAL` em vez de `BIGINT GENERATED BY DEFAULT AS IDENTITY` por ser idiomatico em Postgres e amplamente suportado por Flyway/Hibernate
- `TIMESTAMP WITH TIME ZONE` (ou `timestamptz`) por ser o padrao recomendado pra evitar bugs de fuso
- Sem acento nos comentarios SQL pra evitar dependencia de encoding do driver/locale do banco; documentacao em portugues sem acento e aceitavel aqui

### Tarefa 3 — Adicionar teste de verificacao do Flyway

**Caminho:** `src/test/java/com/laboratorio/financas/shared/FlywayMigrationIT.java`

**Conteudo:** classe estendendo `AbstractIntegrationTest`. Injeta `JdbcTemplate` (ou usa `DataSource` direto) e roda uma assertion confirmando que `flyway_schema_history` contem entrada da V1 com `success = true`.

Esqueleto sugerido (ajustar imports/estilo conforme padrao do projeto):

```java
package com.laboratorio.financas.shared;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.jdbc.core.JdbcTemplate;

import static org.assertj.core.api.Assertions.assertThat;

class FlywayMigrationIT extends AbstractIntegrationTest {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    @Test
    void deveTerAplicadoMigrationV1ComSucesso() {
        // Given - Flyway aplicou as migrations no startup do contexto Spring

        // When
        Boolean v1Aplicada = jdbcTemplate.queryForObject(
            "SELECT success FROM flyway_schema_history WHERE version = '1'",
            Boolean.class
        );

        // Then
        assertThat(v1Aplicada).isTrue();
    }

    @Test
    void deveTerCriadoTabelaHealthcheckComLinhaPlaceholder() {
        // When
        Integer total = jdbcTemplate.queryForObject(
            "SELECT COUNT(*) FROM __healthcheck",
            Integer.class
        );

        // Then
        assertThat(total).isGreaterThanOrEqualTo(1);
    }
}
```

Naming `IT` no final segue convencao Maven Failsafe pra integration tests. Como o projeto usa `mvnw verify` e Testcontainers ja esta configurado, esse teste roda automaticamente no ciclo do `verify`.

Se Spring Boot Test ainda nao esta com `JdbcTemplate` no autoconfigure default deste projeto, o `@SpringBootTest` herdado do `AbstractIntegrationTest` ja traz. Confirmar antes de assumir.
